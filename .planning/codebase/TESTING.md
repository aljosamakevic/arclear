# Testing Patterns

**Analysis Date:** 2026-07-22

## Scope Note

Testing spans two independent stacks that are cross-validated against each other:
- **TypeScript** (`test/`): Vitest + fast-check property-based tests over the SDK
- **Solidity** (`contracts/test/`): Foundry unit tests, revert-matrix tests, fuzz tests
- **Cross-stack parity**: a shared JSON fixture (`test/fixtures/digest.json`) locks EIP-712 digest encoding between viem (TS) and OpenZeppelin's `EIP712` (Solidity) â€” see `test/genFixture.ts` and `contracts/test/DigestParity.t.sol`

When changing anything touching EIP-712 domain/types/digest computation, both suites must be re-verified, and the fixture must be regenerated (`npm run fixture`) before running `forge test`.

## Test Framework

**Runner (TypeScript):**
- Vitest 2.1 (`vitest.config.ts`)
- Config: `vitest.config.ts` â€” includes `test/**/*.test.ts`, excludes `contracts/**` and `node_modules/**`

**Runner (Solidity):**
- Foundry (`forge`), config: `contracts/foundry.toml`
- `solc_version = "0.8.26"`, `via_ir = true`, `optimizer_runs = 1000000`
- Fuzz runs configured globally: `[fuzz] runs = 512`
- `fs_permissions = [{ access = "read", path = "../test/fixtures" }]` â€” grants Forge test read access to the TS-generated fixture directory (required for `DigestParity.t.sol`)

**Assertion Library:**
- TypeScript: Vitest's built-in `expect` (`toBe`, `toEqual`, `toHaveLength`, `toMatch`, `rejects.toThrow`)
- Solidity: `forge-std`'s `Test` base contract â€” `assertEq`, `vm.expectRevert`, `vm.expectEmit`

**Property-based testing:**
- `fast-check` (`fc`) 3.22, used exclusively in `test/netting.test.ts` for algebraic invariants over the netting engine

**Run Commands:**
```bash
npm test                    # vitest run â€” all TS tests (16 property tests + EIP-712 suite)
npm run test:contracts      # cd contracts && forge test -vvv â€” all Solidity tests (26 tests)
npm run fixture              # tsx test/genFixture.ts â€” regenerate test/fixtures/digest.json
```
No coverage command is configured for either stack.

## Test File Organization

**Location (TypeScript):**
- Centralized under `test/`, NOT co-located with `src/` â€” e.g. `test/netting.test.ts` tests `src/netting.ts`, `test/eip712.test.ts` tests `src/iou.ts` + `src/round.ts` together (grouped by protocol concern, not 1:1 with source files)
- Shared fixture data lives in `test/fixtures/digest.json`, generated by the non-test script `test/genFixture.ts`

**Location (Solidity):**
- `contracts/test/` mirrors `contracts/src/` file naming with `.t.sol` suffix: `ClearingHub.t.sol` for `ClearingHub.sol`
- Dedicated fuzz test file kept separate from the unit test file: `ClearingHubFuzz.t.sol`
- Cross-stack parity test isolated in its own file: `DigestParity.t.sol`
- Shared test infrastructure (mock token, actor setup, signature helpers) factored into an abstract base contract under `contracts/test/utils/RoundBuilder.sol`, which both `ClearingHub.t.sol` and `ClearingHubFuzz.t.sol` extend

**Naming:**
- TypeScript: `<concern>.test.ts`
- Solidity: `test_<behavior>` for standard cases, `test_revert_<reason>` for negative/revert cases, `testFuzz_<property>` for fuzz tests

## Test Structure

**Suite organization (Vitest):**
```typescript
describe("netting engine properties", () => {
  it("deltas always sum to zero", () => {
    fc.assert(
      fc.property(arbIous, (ious) => {
        const r = net(ious, { now: NOW });
        expect(r.deltas.reduce((a, b) => a + b, 0n)).toBe(0n);
      }),
    );
  });
  // ...
});
```
(`test/netting.test.ts:45-53`)

- One `describe` block per test file, named after the concern under test (`"netting engine properties"`, `"EIP-712 sign/verify"`)
- Property tests wrap the whole assertion body in `fc.assert(fc.property(...))`
- Example-based tests (specific edge cases) are plain `it(...)` blocks with concrete fixture data, interleaved with property tests in the same `describe` â€” see `"already-settled ids are excluded"` (`test/netting.test.ts:92-98`) as a concrete-example test sitting among property tests

**Suite organization (Foundry):**
```solidity
contract ClearingHubTest is RoundBuilder {
    bytes32 internal constant MANIFEST = keccak256("manifest-1");

    function setUp() public {
        _setUpActors();
    }

    // ---------------------------------------------------------------- deposits
    function test_depositAndWithdraw() public { ... }
    ...
    // ------------------------------------------------------------ happy round
    ...
    // ------------------------------------------------------------ revert matrix
    ...
    // ---------------------------------------------------------- digest parity
}
```
(`contracts/test/ClearingHub.t.sol`)

- Test contract extends a shared abstract harness (`RoundBuilder`), never `Test` directly, when actor/signature setup is needed
- `setUp()` calls into harness helper (`_setUpActors()`) rather than duplicating actor derivation
- Tests are visually grouped with `// --- section name ---` banner comments (fixed-width dashes) separating deposits / happy path / revert matrix / digest parity sections â€” follow this section-comment convention when adding new test groups
- Negative/"revert matrix" tests are grouped together, each testing exactly one failure mode

## Mocking

**TypeScript:**
- No mocking framework (no `vi.mock`, no jest-style mocks) â€” the SDK is tested with real cryptographic primitives (real EIP-712 hashing/signing via viem, real `privateKeyToAccount` test keys) rather than mocked crypto
- Test IOUs that don't need real signatures use a `fakeIou()` helper that fabricates a plausible id via `keccak256(toHex(...))` and a dummy `signature: "0x"`, documented inline: `/** Test IOUs don't need real signatures â€” the engine keys on \`id\`. */` (`test/netting.test.ts:15-31`) â€” use this pattern when a test only exercises id/dedup/expiry logic and doesn't need signature verification
- Fixed test accounts derived from known private keys (`"0x" + "11".repeat(32)`) rather than random â€” deterministic across runs (`test/eip712.test.ts:22-23`)

**Solidity:**
- `MockUSDC` â€” a minimal `ERC20` override (6 decimals, public `mint`) defined in `contracts/test/utils/RoundBuilder.sol:8-18` â€” stands in for the real USDC/EURC token in all contract tests
- Actors are derived deterministically from `keccak256(abi.encode("arclear-actor", i))` as private keys, then sorted ascending by address (mirroring the contract's ascending-participant requirement) â€” see `_setUpActors()` (`contracts/test/utils/RoundBuilder.sol:29-52`)
- No mocking of the contract under test itself â€” tests always exercise the real `ClearingHub` bytecode; `DigestParity.t.sol` uses `vm.deployCodeTo` to deploy the real contract at a fixed address matching the TS fixture, not a mock

**What NOT to mock:**
- Cryptographic operations (hashing, signing, EIP-712 encoding) are never mocked in either stack â€” parity between TS and Solidity encoding is exactly what's being tested, so faking it would defeat the purpose

## Fixtures and Factories

**Shared cross-stack fixture:**
- `test/fixtures/digest.json`, generated by `test/genFixture.ts`, consumed by both `test/eip712.test.ts` (Vitest) and `contracts/test/DigestParity.t.sol` (Foundry via `vm.readFile` + `vm.parseJson*`)
- Regeneration is a manual step (`npm run fixture`) â€” not run automatically before tests; if EIP-712 domain/types change, the fixture must be regenerated or `DigestParity.t.sol` will fail against stale data

**TypeScript factories:**
```typescript
function fakeIou(debtor: Address, creditor: Address, amount: bigint, nonce: bigint, expiry = FUTURE): SignedIou { ... }
function iou(debtor: Address, creditor: Address, amount: bigint, nonce = 1n): Iou { ... }
```
Small local factory functions defined at the top of each test file, not shared across files. Deterministic addresses via a generated array: `Array.from({ length: 6 }, (_, i) => \`0x${(i+1).toString(16).padStart(40,"0")}\`)` (`test/netting.test.ts:10-13`).

**fast-check arbitraries:**
```typescript
const arbIou = fc
  .record({ d: fc.integer({ min: 0, max: 5 }), c: fc.integer({ min: 0, max: 5 }),
    amount: fc.bigInt({ min: 1n, max: 10_000_000n }), nonce: fc.bigInt({ min: 0n, max: 1_000n }) })
  .filter(({ d, c }) => d !== c)
  .map(({ d, c, amount, nonce }) => fakeIou(ADDRS[d], ADDRS[c], amount, nonce));

const arbIous = fc.array(arbIou, { minLength: 0, maxLength: 200 });
```
(`test/netting.test.ts:33-43`) â€” arbitraries are built up compositionally (`fc.record` â†’ `.filter` â†’ `.map` â†’ `fc.array`) and named `arb<Thing>`/`arb<Things>` (plural for arrays).

**Solidity factories:**
- `_simpleRound()` returns a canned 3-participant, zero-sum round used across most `ClearingHub.t.sol` tests (`contracts/test/ClearingHub.t.sol:52-61`)
- `_buildSignatures(nonce, participants, deltas, manifestHash)` signs a full round for all participants using their known keys, returning `bytes[] memory sigs` ready to pass to `executeRound`

## Test Types

**Unit Tests:**
- Solidity: `ClearingHub.t.sol` â€” deposits/withdrawals, happy-path round execution, full revert matrix (12 distinct negative cases covering nonce mismatch, unsorted/duplicate participants, length mismatch, too few participants, non-zero sum, bad signature, tampered delta/manifest, insufficient collateral, withdraw-front-runs-execution, cross-hub replay, pause gating)

**Property Tests:**
- TypeScript: `test/netting.test.ts` â€” 8 properties over the netting engine: zero-sum output, shuffle-invariance (determinism regardless of input order), dedup idempotence (multiset == set), expired-IOU irrelevance, ascending-participant ordering, settled-volume bounds, plus 2 concrete example tests (already-settled exclusion, zero-net-but-consumed participant retention, perfect circular flow = 100% compression)

**Fuzz Tests (Solidity):**
- `ClearingHubFuzz.t.sol` â€” `testFuzz_roundExecution(uint256 seed)`: builds a random zero-sum flow matrix among 5 actors, funds debtors exactly, executes, and asserts per-account collateral deltas match and hub token balance is conserved
- `testFuzz_perturbationAlwaysReverts(uint256 seed, bool tamperSig)`: asserts that any single-byte signature tamper OR any zero-sum-preserving delta nudge causes a full revert with untouched state â€” the core "unanimous consent" security property

**Cross-Stack Parity Test:**
- `contracts/test/DigestParity.t.sol` â€” reads the shared fixture, redeploys `ClearingHub` at a fixed address via `vm.deployCodeTo`, recomputes `hashRound` on-chain, and asserts it equals the viem-computed digest from the fixture; also asserts the viem-produced signature recovers to the expected signer via `ECDSA.recover`. This is the single test that guarantees the TS SDK and Solidity contract agree on EIP-712 encoding byte-for-byte.

**E2E Tests:**
- `demo/e2e.ts` (run via `npm run e2e:anvil` / `npm run e2e:testnet`) is a manual/scripted end-to-end flow (5 agents sign ~100 IOUs, net, execute one round on a live anvil instance or Arc Testnet) â€” not part of the automated test suite (`npm test` / `npm run test:contracts`), used for demoing and manual verification instead.

## Common Patterns

**Async testing (Vitest):**
```typescript
it("rejects a signer that is not the debtor", async () => {
  await expect(
    signIou(HUB, iou(alice.address, bob.address, 42n), bob),
  ).rejects.toThrow(/not debtor/);
});
```
(`test/eip712.test.ts:43-47`) â€” async rejection assertions use `rejects.toThrow(/regex/)` matching a substring of the thrown `Error` message rather than an exact string.

**Error/revert testing (Foundry):**
```solidity
function test_revert_wrongNonce() public {
    (address[] memory p, int256[] memory d) = _simpleRound();
    bytes[] memory sigs = _buildSignatures(7, p, d, MANIFEST);
    vm.expectRevert(abi.encodeWithSelector(ClearingHub.WrongRoundNonce.selector, 0, 7));
    hub.executeRound(7, p, d, MANIFEST, sigs);
}
```
Prefer `vm.expectRevert(abi.encodeWithSelector(Contract.CustomError.selector, args...))` when the error carries diagnostic parameters worth asserting; use bare `vm.expectRevert()` only when the exact error/args aren't the point of the test (e.g. `test_pause_onlyOwner`, `test_revert_whenPaused`).

**Event assertion (Foundry):**
```solidity
vm.expectEmit(true, true, false, true);
emit ClearingHub.RoundExecuted(0, _digest(0, p, d, MANIFEST), MANIFEST, 3, 3e6);
hub.executeRound(0, p, d, MANIFEST, sigs);
```
Standard Foundry `expectEmit` + re-declare-the-expected-event pattern (`contracts/test/ClearingHub.t.sol:71-73`).

**Invariant/property assertions (both stacks):**
- TypeScript property tests always assert a mathematical invariant (sums to zero, equals recomputation, order-independence) rather than a fixed expected value â€” prefer this style for any new function with an algebraic property (idempotence, commutativity, conservation)
- Solidity fuzz tests assert conservation invariants with descriptive failure messages as the last `assertEq` argument: `assertEq(usdc.balanceOf(address(hub)), hubBalanceBefore, "hub balance not conserved");` â€” always include a message string on invariant-style assertions so fuzz failures are self-explanatory.
